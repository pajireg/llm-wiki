# Write-Time Keyword Expansion for Search

**Date**: 2026-05-21
**Status**: Approved, pending implementation
**Author**: sumin

## Problem

The auto-injection hook (v0.6.0+) and `/llm-wiki:ask` rely on a SQLite FTS5
keyword index. Search is BM25 keyword-only: if the user's prompt shares no
literal tokens with a page, that page is never surfaced — even when it is
conceptually the right answer. Comparison project
[seCall](https://github.com/hang-in/seCall) solves this with vector embeddings
(Ollama BGE-M3 + HNSW), but that requires an embedding model resident at query
time (every prompt), which conflicts with this plugin's core identity:

- Python 3.9 **stdlib only**, no external dependencies.
- No daemon, no network call on the hot path.
- The injection hook **must never block the prompt** (every error path exits 0).

## Goal

Recover much of the "conceptually related, not just literally matching" benefit
**without** any new dependency, daemon, or per-prompt model call.

## Approach: LLM expansion at write time

The embedder we already have is Claude itself, running during `/llm-wiki:ingest`.
Instead of embedding at query time, we **expand the document side at write time**:
when synthesizing a page, Claude generates alternate names and related concept
terms and stores them in frontmatter. The FTS5 index ingests them as additional
columns. At query time the hook stays exactly as today — pure stdlib FTS5, OR of
prompt tokens — but now matches against a richer document surface, so
synonym/abbreviation/concept queries hit.

This is not true semantic distance. It captures synonyms, abbreviations,
alternate names, language variants, and adjacent concepts. True vector search
remains a future opt-in layer (out of scope).

## Data model

Two **optional** frontmatter fields on wiki pages (not required → lint must not
fail when absent):

```yaml
aliases:  [alternate names, abbreviations, English variants]   # high precision
keywords: [related concepts / topic terms]                     # recall expansion
```

- Generated by `/llm-wiki:ingest` during synthesis. Claude produces 5–15 terms
  in the **user's primary language + English variants** (not hardcoded
  Korean-English — the plugin is public and language-agnostic).
- `aliases` = other ways to name the same thing (high-precision signal).
- `keywords` = adjacent concepts / topic terms (recall signal).
- Frontmatter remains the source of truth; the index only reads these fields.

## Index schema (`scripts/rebuild-index.py`)

Add two columns to the FTS5 virtual table:

```sql
CREATE VIRTUAL TABLE pages_fts USING fts5(
  id UNINDEXED,
  title, summary, body, aliases, keywords,
  tokenize='unicode61 remove_diacritics 2'
);
```

- `rebuild-index.py` does **not** call an LLM. It copies the `aliases`/`keywords`
  frontmatter that `/ingest` already wrote into the index. Both fields are
  flattened to a space-joined string for indexing; missing → empty string.
- The `pages` metadata table is unchanged (these fields are search-only, not
  displayed).

### Migration

FTS5 cannot add columns to an existing virtual table. The index is a derived
artifact (safe to wipe — already documented). On open, detect the old schema
(missing `aliases` column via `PRAGMA table_info` / a probe query) and trigger a
full **wipe & rebuild**, reusing the existing self-healing path. Cost: one index
rebuild on the first run after upgrade.

## Search ranking (`hooks/user-prompt-inject.py`)

- The FTS5 `MATCH` already searches **all** indexed columns, so the new columns
  are matched automatically — the matching logic needs no change.
- Switch ranking from `ORDER BY f.rank` (uniform BM25) to an explicit
  `bm25(pages_fts, ...)` with per-column weights:
  - `aliases` weighted highest (exact alternate-name match is a strong signal),
  - `title` high, `summary`/`keywords` medium, `body` baseline.
  - The `id UNINDEXED` column still takes a weight slot (0.0) in column order.
  - Weights are tunable constants; start conservative and adjust against the
    test fixtures.
- Query side is unchanged: stdlib OR-of-tokens, no embedding, no network.
  `/llm-wiki:ask` shares the same index and benefits identically.

## Command & schema documentation

- `commands/ingest.md` and `skills/llm-wiki/SKILL.md`: add the instruction to
  generate `aliases` + `keywords` (user language + English) for each synthesized
  page.
- `templates/schema/`: document the two optional fields. New vaults receive them
  via `/init`; existing vaults via `/upgrade-schema`. Because the fields are
  optional, vaults without the updated schema still function — `/ingest` knows
  to generate them regardless of whether the schema doc mentions them.

## Backfill: lazy

No backfill command. Existing pages have no `aliases`/`keywords` and keep
matching on title/summary/body as before. They gain the new fields naturally the
next time they are touched by `/ingest`. Minimal scope; large existing vaults
simply improve incrementally.

## Testing

- Index populates `aliases`/`keywords` columns from frontmatter and they
  participate in search results.
- Old-schema DB (no `aliases` column) triggers automatic wipe & rebuild on open.
- Pages lacking `aliases`/`keywords` (lazy/legacy) still index and search without
  error — confirms backward compatibility.
- Existing 15 tests continue to pass; ranking change does not break the
  injection-hook fixtures.

## Out of scope (future)

- True vector embedding search as an **opt-in** layer (Ollama / ONNX / API),
  hybrid BM25 + vector RRF — only when an embedding backend is configured.
- Multi-agent ingest (Codex, Gemini, ChatGPT) — future goal.
- Version bump decision deferred to post-implementation (keep release cadence
  slow per author preference).
